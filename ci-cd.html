CI/CD

Apr 02, 20262 min read

Automatizzare build, test e deploy per rilasciare con fiducia

Cos’è CI/CD

Continuous Integration — integrare e testare il codice frequentemente Continuous Delivery — il codice è sempre pronto per il deploy Continuous Deployment — deploy automatico ad ogni merge obiettivo — ridurre il rischio di ogni rilascio rilasciando spesso “se fa male, fallo più spesso” — Jez Humble

Pipeline CI

trigger — push o pull request su un branch checkout del codice installazione delle dipendenze linting e formatting check type checking (TypeScript) unit test e integration test build dell’applicazione analisi della sicurezza delle dipendenze

Pipeline CD

build dell’immagine Docker push al container registry deploy su ambiente di staging smoke test sull’ambiente di staging approvazione manuale (per Continuous Delivery) deploy su produzione notifica del risultato

Ambienti

development — locale, ogni developer staging — replica della produzione per test production — ambiente reale feature environments — ambienti effimeri per ogni PR environment parity — staging deve essere identico a production

GitHub Actions in dettaglio

workflow YAML — on, jobs, steps runner — ubuntu-latest, windows-latest, macos-latest self-hosted runner — runner su infrastruttura propria artifact — file prodotti da un job e passati ad altri job cache — dipendenze tra run diverse environment secrets — secrets specifici per ambiente

Strategie di Deploy

rolling update — rimpiazza gradualmente le istanze vecchie blue-green — due ambienti identici, switch del traffico canary — rilascio graduale a una percentuale di utenti feature flags — separare deploy da release

Rollback

cos’è un rollback — tornare alla versione precedente rollback automatico su health check fallito database migrations e rollback — il problema più complesso forward fix vs rollback — quando preferire uno all’altro

Qualità nella Pipeline

fail fast — bloccare la pipeline al primo errore test paralleli — ridurre il tempo della pipeline coverage threshold — bloccare se la coverage scende sotto una soglia performance budget — bloccare se le metriche peggiorano dependency audit — bloccare se ci sono vulnerabilità note
